#### 2️⃣ **文件处理模块（vector/service）**

这是整个系统最核心的部分。

**TextExtractionService - 文本提取**
```java
// 支持4种格式
PDF/Word/Excel → Apache Tika提取
Markdown/TXT → 直接读取UTF-8
```

**TextChunkService - 智能分片**
```java
// 4种分片策略
1. SEMANTIC（语义分块）- 推荐
   按段落和标题等语义边界切分，保持语义完整性

2. FIXED_SIZE（固定大小）
   按字符数切分，带重叠窗口

3. PARAGRAPH（段落）
   按双换行符切分

4. SENTENCE（句子）
   按中英文句号切分
```

**HybridVectorService - 混合向量化与检索**
```java
// 双重向量化流程
1. 稠密向量：使用text-embedding-v3生成1024维稠密向量
2. 稀疏向量：使用BM25Encoder生成稀疏向量
3. 存储：将稠密向量和稀疏向量一同存储到向量数据库
```

**示例：**
```
用户上传100页PDF设计文档
  ↓
TextExtractionService提取出20万字文本
  ↓
TextChunkService按语义分块成400个分片（每片约500字）
  ↓
HybridVectorService双重向量化：
  - 稠密向量：text-embedding-v3生成
  - 稀疏向量：BM25Encoder生成
  ↓
存储到向量数据库（每个分片同时存储稠密+稀疏向量）
```

#### 3️⃣ **记忆系统模块（memory）**

**短期记忆（ShortTermMemoryService）**
```
实现方式: Redis List + 字符数控制 + 滑动窗口 + 自动摘要

数据结构:
Key: stm:{userId}:{sessionId}
Value: List of JSON objects { role, content }

字符数控制逻辑:
1. 每次添加新消息后，计算总字符数
2. 如果总字符数超过32k，触发摘要机制
3. 摘要策略：(超出32k的部分 + 窗口内前一半) → 调用LLM生成摘要
4. 保留：[摘要文本] + [窗口内的后一半内容]

作用:
用户: "设计规范是什么？"
系统: "包括色彩、排版、图标..."
用户: "那色彩部分呢？"  ← 系统知道"那"指设计规范
```

**长期记忆（LongTermMemoryService）**
```
实现方式: MySQL + 向量化存储 + 语义召回

存储内容:
- 用户偏好（"我喜欢简洁的设计"）
- 重要查询历史
- 系统自动提取的关键信息

召回逻辑:
1. 将用户查询向量化
2. 在向量数据库中搜索相似记忆
3. 按语义相似度排序
4. 返回最相关的3条记忆

作用:
记住用户说过的话，提供个性化回答
```

#### 4️⃣ **检索系统模块（vector/service）**

**HybridVectorService - 混合检索**
```java
检索流程:
1. 向量语义检索（稠密向量）
   - 将查询向量化(text-embedding-v3)
   - 计算余弦相似度
   - 返回最相似的分片

2. BM25关键词检索（稀疏向量）
   - 将查询转为BM25稀疏向量
   - 在向量数据库中进行关键词匹配
   - 计算BM25分数

3. Pinecone内部自动融合
   - 同时利用稠密向量（语义）和稀疏向量（关键词）匹配
   - 返回综合相似度最高的结果
```

#### 5️⃣ **RAG问答模块（rag）**

**完整RAG流程（RAGService.java）**
```java
public RAGResultDTO query(queryDTO, userId, sessionId) {
    // 第1步：查询重写
    rewrittenQuery = rewriteQuery(query, sessionId, userId);
    // 基于短期记忆上下文，补全问题
    
    // 第2步：混合检索
    knowledgeChunks = hybridVectorService.hybridSearch(rewrittenQuery);
    // 稠密+稀疏向量检索
    
    // 第3步：检索长期记忆
    longTermMemories = longTermMemoryService.retrieveMemories(userId, rewrittenQuery);
    // 用户历史偏好
    
    // 第4步：构建Prompt
    prompt = systemPrompt + 
             "\n检索到的知识:\n" + knowledgeChunks +
             "\n用户历史记忆:\n" + longTermMemories +
             "\n问题: " + rewrittenQuery;
    
    // 第5步：调用LLM
    answer = llmService.generateSummary(prompt);
    
    // 第6步：更新记忆
    shortTermMemoryService.addMessage(userId, sessionId, "user", query);
    shortTermMemoryService.addMessage(userId, sessionId, "assistant", answer);
    longTermMemoryService.extractAndStoreMemories(userId, query, answer);
    
    return new RAGResultDTO(query, answer, knowledgeChunks);
}
```

## 第六部分：核心技术亮点

### ⭐ 1. 混合检索（Dense + Sparse Vectors）
```
传统向量检索的缺点:
- 语义匹配能力强，但对专有名词、精确匹配效果差

BM25的优点:
- 精确关键词匹配
- 对专有名词敏感

text-embedding-v3的优势:
- 同时支持稠密向量（语义）和稀疏向量（关键词）
- 单一模型双份收益

混合检索 = 稠密向量语义 + 稀疏向量关键词
→ 既能理解语义，又能精确匹配
```

### ⭐ 2. 查询重写（Context Rewriting）
```
用户习惯用省略语:
第一轮: "设计规范是什么？"
第二轮: "那色彩部分呢？"

系统通过短期记忆理解上下文:
自动重写为: "设计规范中的色彩部分是什么？"
→ 大幅提高检索准确率
```

### ⭐ 3. 字符数控制的滑动窗口
```
传统的token数限制复杂，难以直观理解
改用字符数限制（32k字符）：
- 更直观，易于理解和调试
- 避免不同模型tokenizer差异带来的问题
- 摘要触发机制更稳定
```

### ⭐ 4. 记忆系统
```
短期记忆: 让系统"记住"当前对话
长期记忆: 让系统"了解"用户偏好
→ 提供个性化的智能回答
```
